# Rules

A **rule** is the fundamental unit in Run-IQ. It declares a calculation to perform, when to apply it, and with what parameters.

## Rule Structure

```typescript
interface Rule {
  id: string;                    // Unique identifier
  version: number;               // Rule version number
  model: string;                 // Calculation model ("FLAT_RATE", "PROGRESSIVE_BRACKET", ...)
  params: unknown;               // Model-specific parameters
  condition?: Expression;        // When to apply this rule
  priority: number;              // Dominance ordering (higher = evaluated first)
  dominanceGroup?: string;       // Conflict group (e.g., "TVA", "IS")
  effectiveFrom: Date;           // Rule is active from this date
  effectiveUntil: Date | null;   // Rule expires after this date (null = no expiry)
  tags: readonly string[];       // Labels for filtering
  checksum: string;              // SHA-256 of serialized params
  schemaVersion?: string;        // Optional schema version
}
```

## Key Fields

### `model` ‚Äî What to Calculate

The `model` field is a string key that maps to a registered `CalculationModel`. The engine itself doesn't know what `"FLAT_RATE"` means ‚Äî it looks it up in the `ModelRegistry`, which was populated by a plugin.

Available models depend on which plugins are loaded. The fiscal plugin provides:

| Model | Description |
|---|---|
| `FLAT_RATE` | `base * rate` |
| `PROGRESSIVE_BRACKET` | Cumulative bracket calculation |
| `MINIMUM_TAX` | `max(base * rate, minimum)` |
| `THRESHOLD_BASED` | Applies only above a threshold |
| `FIXED_AMOUNT` | Returns a constant value |
| `COMPOSITE` | Combines sub-models (SUM, MAX, MIN) |

### `params` ‚Äî Model Parameters

Each model defines its own parameter schema. The engine passes `params` to the model without inspecting it ‚Äî the model validates and uses them.

```typescript
// FLAT_RATE params
{ rate: 0.18, base: "revenue" }

// PROGRESSIVE_BRACKET params
{
  base: "taxableIncome",
  brackets: [
    { from: 0, to: 500_000, rate: 0 },
    { from: 500_000, to: 1_000_000, rate: 0.10 },
    { from: 1_000_000, to: null, rate: 0.25 },
  ]
}

// COMPOSITE params
{
  aggregation: "SUM",
  steps: [
    { model: "FLAT_RATE", params: { rate: 0.07, base: "grossSalary" } },
    { model: "FLAT_RATE", params: { rate: 0.04, base: "grossSalary" } },
  ]
}
```

### `dominanceGroup` ‚Äî Smart Conflict Resolution

The `dominanceGroup` (often mapped to a `category` like "TVA" or "CNSS") defines the scope of a conflict. 

When multiple rules share the same **priority**, the engine checks their group:
- **Different Groups**: The rules are considered independent and are both applied (additive behavior).
- **Same Group**: The rules are in conflict. The engine keeps the first one and skips the others (or throws in strict mode).

This allows you to have a TVA rule and an IS rule both at priority `3000` without them fighting each other.

### `priority` ‚Äî Dominance Order

Priority determines evaluation order (higher = first).

<Callout type="info" title="Sovereign Priority">
  If you provide an explicit `priority` in the rule JSON (e.g., `priority: 1234`), the engine will respect it exactly. Automatic calculation (such as by jurisdiction) only triggers if the priority field is missing or null.
</Callout>

Combined with `dominanceGroup`, it provides granular control over which rule wins in a specific category.

<Callout type="tip" title="Additive by default">
  If you don't provide a `dominanceGroup`, the engine treats each rule as its own unique group, allowing multiple rules to coexist at the same priority level.
</Callout>

### `condition` ‚Äî When to Apply

A rule's condition is evaluated by a DSL evaluator. The condition contains:

```typescript
interface Expression {
  dsl: string;       // Which evaluator to use ("jsonlogic", "cel", ...)
  value: unknown;    // The expression itself (DSL-specific format)
}
```

The engine passes the condition to the DSL evaluator registered under `dsl`. If the evaluator returns `true`, the rule is applied.

```typescript
// JSONLogic ‚Äî apply if revenue > 0
{
  dsl: "jsonlogic",
  value: { ">": [{ "var": "revenue" }, 0] }
}

// JSONLogic ‚Äî apply if employee type is "permanent"
{
  dsl: "jsonlogic",
  value: { "==": [{ "var": "employeeType" }, "permanent"] }
}
```

<Callout type="info" title="DSL agnostic">
  The engine core never parses condition values. It delegates entirely to the
  registered DSL evaluator. You can use JSONLogic, CEL, or any custom DSL ‚Äî just
  register the evaluator.
</Callout>

### `priority` ‚Äî Dominance Order

When multiple rules match, priority determines evaluation order. Higher priority rules are evaluated first. If two rules have the same priority and `strict` mode is enabled, the engine throws a `RuleConflictError`.

### `effectiveFrom` / `effectiveUntil` ‚Äî Temporal Filtering

Rules are only considered if the evaluation date falls within their `[effectiveFrom, effectiveUntil]` window. Set `effectiveUntil` to `null` for rules with no expiry. This enables versioning ‚Äî you can have two versions of the same tax rule with different effective periods, and the engine will automatically select the correct one.

### `checksum` ‚Äî Total Identity Security

The checksum is a SHA-256 hash that acts as a tamper-evident seal for the rule's entire logic.

<Callout type="info" title="Total Identity Security">
  Unlike simple data hashes, a Run-IQ checksum covers the rule's **complete identity**: the `model`, the `params`, the `condition`, and the `priority`. Any change to how the rule decides or calculates will invalidate the checksum.
</Callout>

<Callout type="info" title="Canonical Hashing">
  The engine uses **canonical serialization**: all keys in the hashed object are recursively sorted alphabetically before hashing. This makes the checksum robust against different property orders in the JSON input.
</Callout>

At evaluation time, the engine re-computes this identity hash from the provided rules and compares it against the stored `checksum`. If they don't match, the rule is skipped (or an error is thrown in strict mode).

### `tags` ‚Äî Filtering Labels

Tags allow the host application to filter rules before evaluation. Pass tags in `input.meta.tags`.

<Callout type="tip" title="Environmental Filtering">
  For environmental filters like `country` or `currency`, it is best practice to use **meta.context**. The Fiscal Plugin, for example, expects `meta.context.country` to selectively activate rules.
</Callout>

```typescript
// Only evaluate rules tagged "irpp" or "tva"
const input = {
  data: { revenue: 5_000_000 },
  requestId: 'eval-001',
  meta: {
    tenantId: 'my-company',
    tags: ['irpp', 'tva'],
    context: { country: 'TG' }
  },
};
await engine.evaluate(rules, input);
```

## Rule Lifecycle

```
Rule created ‚Üí checksum computed ‚Üí stored in database
                                          ‚Üì
Engine.evaluate() called ‚Üí rules loaded ‚Üí filtered by date/tags
                                          ‚Üì
                             conditions evaluated by DSL
                                          ‚Üì
                          matching rules sorted by priority
                                          ‚Üì
                          each rule's model.calculate() called
                                          ‚Üì
                        results aggregated ‚Üí trace built ‚Üí snapshot saved
```
