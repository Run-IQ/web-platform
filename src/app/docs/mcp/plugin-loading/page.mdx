# MCP Plugin Loading

The MCP server can load plugin bundles at runtime, enabling domain extensibility without recompiling. It supports both local directory scanning and **automatic NPM installation**.

## Loading Methods

### 1. Remote NPM Plugins (Recommended)

The server can automatically download and install plugins from NPM. These are stored persistently in your user home directory to ensure they are available across different execution contexts (like Claude Desktop).

```bash
npx @run-iq/mcp-server --plugin @run-iq/plugin-fiscal
```

**How it works:**
1. **Resolve** — Checks if the plugin is installed locally or in the global cache (`~/.run-iq/mcp-plugins`).
2. **Install** — If missing, runs `npm install --no-save` into the global cache directory.
3. **Import** — Dynamically loads the bundle into the engine.

### 2. Local Directory Scanning

Load multiple plugins from a local folder. Useful for development or private plugins.

```bash
npx @run-iq/mcp-server --plugins-dir ./my-plugins/
```

1. **Scan** — Reads the directory for `.js` and `.mjs` files.
2. **Import** — Dynamically imports each file.
3. **Validate** — Checks for the `PluginBundle` shape.

<Callout type="info" title="Global Cache Location">
  Plugins installed via `--plugin` are cached in `~/.run-iq/mcp-plugins` (Windows: `C:\Users\<User>\.run-iq\mcp-plugins`). This ensures they persist even if the server is launched from different directories.
</Callout>

## Creating a Loadable Bundle

A bundle file must export an object matching the `PluginBundle` interface. The server is flexible and accepts `export default`, named exports, or bundles nested under a `bundle` property (common in compiled CJS/ESM modules).

```typescript
// my-plugins/insurance.mjs (or .js)
import { InsurancePlugin } from './InsurancePlugin.js';
import { insuranceDescriptor } from './descriptor.js';

export default {
  plugin: new InsurancePlugin(),
  descriptor: insuranceDescriptor,
  // Optional: include domain-specific DSL evaluators
  // dsls: [new CustomDSL()],
};
```

### PluginBundle Interface

```typescript
interface PluginBundle {
  plugin: PPEPlugin;
  descriptor: PluginDescriptor;
  dsls?: DSLEvaluator[];
}
```

## DescriptorRegistry

The `DescriptorRegistry` aggregates metadata from all loaded plugin descriptors:

```typescript
class DescriptorRegistry {
  register(descriptor: PluginDescriptor): void;
  getAll(): readonly PluginDescriptor[];
  getRuleExtensions(): readonly RuleFieldDescriptor[];
  getInputFields(): readonly InputFieldDescriptor[];
  getExamples(): readonly RuleExample[];
  isEmpty(): boolean;
}
```

**Key behaviors:**
- `getRuleExtensions()` — Flattens all plugin rule extension fields into a single list. Used by `create_rule` schema and `schema://rules` resource.
- `getInputFields()` — Deduplicates input fields across plugins (first-win by name)
- `getExamples()` — Aggregates all rule examples from all plugins

## Schema Adaptation

When plugins are loaded, the MCP tools and resources automatically adapt:

| Component | Adaptation |
|---|---|
| `list_models` tool | Lists all calculation models and identifies their **source plugin** |
| `create_rule` tool | Dynamic schema includes plugin extension fields |
| `validate_rules` tool | Validates plugin-specific fields |
| `inspect_rule` tool | Checks extension field correctness |
| `schema://rules` resource | Includes extension fields, models, input fields, and examples |
| `analyze-text` prompt | Includes domain guidelines and available fields from loaded plugins |
| `domain-expert` prompt | References loaded domain expertise |
| `create_checksum` tool | Creates a checksum |

<Callout type="info" title="No Recompilation needed">
  Adding a new domain (insurance, payroll, customs) only requires passing the `--plugin` flag with the `NPM package name`.
  You can also pass a local path to a bundle file and this only requires dropping a `*.mjs` or `*.js` file to the plugin directory.
  <br/>The MCP server handles discovery, installation, and engine integration automatically.
</Callout>

## Example: Multi-Domain Setup

```bash
# Directory structure
my-plugins/
├── fiscal.js        # @run-iq/plugin-fiscal bundle
├── payroll.js       # @run-iq/plugin-payroll bundle
└── insurance.mjs    # Custom insurance bundle

# Start MCP server with a mix of local and remote plugins
npx @run-iq/mcp-server --plugins-dir ./my-plugins --plugin @run-iq/plugin-fiscal
```

The LLM now has access to all models, rule extensions, and domain expertise from both sources.
